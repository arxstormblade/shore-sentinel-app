# Shore Sentinel v1.1 SSH-Push Security Decision (Historical)

**Status:** Historical approved policy baseline — superseded for executable SSH transport and deadline details; not a current implementation or deployment authority

> **Supersession notice:** For the current executable SSH transport, staging, and deadline constraints, use [`remote-runner-install.md`](remote-runner-install.md), [`infra/ssh-fixture/fixture-contract.json`](../../infra/ssh-fixture/fixture-contract.json), and the corresponding implementation and contract tests. They define fixed `--stage-request` JSON stdin staging with no SFTP (8192 bytes / 5 seconds) and the root runner's 100-second execution deadline. The SFTP staging timeout and 20-minute scanner execution deadline in this historical decision are not current release authority.

## Decision

Shore Sentinel may support managed-machine SSH push only through a constrained, pinned-host, short-lived execution path. The existing placeholder worker path must not be extended directly.

The browser requests a managed-machine scan and may supply a directory scope. It never receives credentials, SSH host keys, internal execution data, queue payloads, object-store access, or worker endpoints.

## Allowed data flow

```text
Browser
  → same-origin Web proxy
  → API: RBAC, target ID, directory scope only
  → Postgres: immutable request snapshot
  → BullMQ: opaque job/run identifiers only
  → authenticated Node SSH worker
  → one-use API execution-context grant
  → pinned SSH target on an approved CIDR and port
  → signed scanner bundle and fixed remote runner
  → bounded report artifacts through API ingestion
  → private object storage and immutable evidence metadata
  → parser worker and safe browser DTOs
```

## Non-negotiable controls

### Target and network policy

- Targets are canonical hostnames or IP addresses only; reject URLs, userinfo, paths, wildcard DNS, control characters, malformed IDNs/IPs, zone IDs, and invalid ports.
- Resolve immediately before every connection. Connect to the validated numeric address, not a fresh hostname lookup.
- Enforce a tenant-scoped explicit CIDR and SSH-port policy. Default port is `22`; no implicit public-Internet policy exists.
- Always deny loopback, link-local, multicast, unspecified, cloud-metadata, Docker bridge, and control-plane/service networks.
- The SSH worker has dedicated egress enforcement at the deployment firewall layer. Application validation alone is insufficient.

### SSH identity and credentials

- Require an out-of-band verified active `ssh-ed25519` host-key pin per tenant, target, and port. No trust-on-first-use and no shared mutable `known_hosts` file.
- Host-key rotation requires an authorized audited workflow: a verified successor overlaps the retiring key for 14 days; emergency revocation disables SSH scanning and invalidates grants.
- Production API startup fails without a valid secret-mounted master key. There is no development fallback key for production.
- Only the API can decrypt credentials through a versioned AEAD envelope bound to tenant, credential, and credential type.
- The worker receives a credential only in memory via a one-use execution grant with a 60-second TTL. It must never enter Redis, argv, environment variables, logs, events, artifacts, or persistent files.

### Bundle and command integrity

- Build an allowlisted scanner-only bundle from a clean tagged commit. Validate it with the repository scanner validator.
- Generate a canonical manifest with per-file hashes, bundle hash, entry point, Git commit, and output schema version. Sign the manifest with an offline Ed25519 release key; workers carry only verification public keys.
- Verify signature, manifest, bundle bytes, and file map at worker startup and before transfer.
- The remote runner is fixed and signed. User-supplied values never enter shell text. A validated request is staged in a UUID-only owner-private directory and is consumed through structured data.
- `.` resolves server-side to an enrolled absolute default root. The remote runner uses `realpath`, requires an existing directory under an enrolled allowed root, and rejects traversal and symlink escapes.

### Limits and cancellation

| Control | v1.1 secure default |
|---|---:|
| SSH worker concurrency | 1 |
| Connect/auth timeout | 15 seconds each |
| SFTP staging timeout | 60 seconds |
| Scanner execution deadline | 20 minutes |
| Whole run deadline | 25 minutes |
| Captured stdout/stderr | 1 MiB each |
| Raw JSON artifact | 10 MiB |
| Markdown artifact | 5 MiB |
| SARIF artifact | 20 MiB |
| PDF artifact | 25 MiB |
| Total accepted artifacts | 60 MiB |
| Artifact count | one each: JSON, Markdown, SARIF, PDF |

- Cancellation is tenant-scoped, auditable, and terminal. Queued jobs are removed; active work receives an opaque cancellation signal.
- The eventual SSH executor must abort the channel and remote process group, wait up to 10 seconds, then force-close. Until the executor exists, cancellation must not claim remote-process termination.
- Events, retries, success, failures, and artifact ingestion after cancellation must not revive a cancelled run. Late artifacts are quarantined and audited.

### Evidence attribution

- Workers use a service token scoped to worker identity, tenant, run, and action.
- Each run records server-derived requester, target snapshot, host-key fingerprint, network-policy reference, canonical directory scope, worker/image identity, bundle/manifest identities, and lifecycle timestamps. No caller-supplied evidence identity is authoritative.
- The API hashes accepted artifacts itself; workers do not receive object-store credentials.
- Artifact storage is private and retention-protected. Events and audit entries are hash-chained and append-only for the application role.

## Required implementation work

1. Add forward-only schema migrations for host-key pins, egress policies, execution grants, bundle releases, cancellation metadata, and evidence fields.
2. Replace fallback secret sealing with API-only versioned secret-file key handling.
3. Add target policy, root policy, and host-key enrollment/rotation APIs protected by operator/admin permissions.
4. Add internal worker authentication and one-use execution-context grants.
5. Add signed bundle production and verification.
6. Replace placeholder worker output with the constrained SSH executor.
7. Make MinIO private and remove externally published control-plane services from deployment configuration.
8. Add disposable SSH fixture, negative security tests, cancellation race tests, and browser regression coverage.

## Release acceptance gates

The v1.1 release cannot merge or deploy until all of these pass:

- No fallback encryption key, worker token fallback, public artifact bucket, or host-published Redis/Postgres/MinIO/Python-worker service.
- SSH launch fails closed for a missing host pin, root policy, egress policy, enabled credential, trusted bundle, or valid internal grant.
- Host-key mismatch fails before authentication.
- Tests prove directory data cannot reach shell text and remote path escapes are rejected.
- Redis payloads contain no credentials, execution context, or artifact bodies.
- Stream caps reject over-budget data before storage, and accepted artifact hashes verify on retrieval.
- Cancellation remains terminal under queue/worker race conditions.
- A disposable SSH fixture proves the expected success and failure paths; no live customer system is used.
- API, worker, scanner, Web, Compose, authenticated browser QA, Aegis review, and Athena review all pass.
