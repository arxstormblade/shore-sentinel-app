# Shore Sentinel v1.1.0 — Managed Machine Scan Controls and Hardware Summary

**Status:** Approved product design

## Goal

Improve the managed-machine workflow without weakening scan attribution, RBAC, or evidence integrity. Release the result as Shore Sentinel `v1.1.0`.

## Scope

### Clearer navigation and language

- Replace user-facing **Open dossier** text with **Open Machine** in inventory and dashboard machine rows.
- Use sentence case for labels, headings, controls, statuses, and helper text unless a proper noun, acronym, hostname, or operating-system name requires different casing.
- Refine the managed-machine typography hierarchy: readable operational labels, balanced heading scale, tabular figures for metrics/timestamps, and consistent technical naming.

### Hardware summary

Immediately beneath the machine name, display a concise hardware summary when the latest trusted scan data is available:

- operating system/platform;
- logical CPU cores;
- installed memory;
- root/storage capacity.

The client receives only browser-safe summary values. It never receives scanner raw output, runtime internals, credentials, network adapter addresses, artifact storage references, or worker metadata. When no trusted hardware summary is available, show a clear pending state such as **Hardware details appear after the next completed scan**.

### Scan controls

The active-scan region becomes an explicit scan control band:

- **Stop scan** is visible only for a queued, leased, or running managed-machine scan and only to roles allowed to edit scan jobs.
- Stopping is idempotent: it marks the run/job cancelled, timestamps completion, emits an auditable `job.cancelled` event, and returns the final public run DTO.
- A stopped scan preserves any already-created evidence. It never deletes scan history or artifacts.
- Workers must not continue a cancelled queued/leased job. If a worker reports after cancellation, terminal cancellation remains authoritative.

### SSH-push execution boundary

The selected v1.1 execution model is **SSH push**. A managed-machine scan runs the packaged scanner on the enrolled machine, not in the Shore Sentinel API or worker container.

- SSH push is enabled only for managed machines with a pinned host-key fingerprint, an approved SSH credential reference, and an explicitly supported remote runtime (Python 3 plus the required scanner runtime dependencies).
- The API decrypts stored SSH credentials only to create a short-lived, worker-authenticated execution context. Browser clients never receive credentials, host-key material, or internal execution context.
- The worker transfers the versioned scanner bundle over SFTP and invokes it with a fixed command template plus a safely encoded, validated `--target` argument; it must not concatenate user input into shell syntax.
- Existing machines without a pinned host key display a clear prerequisite state and cannot launch an SSH-push scan until an authorized operator saves the verified fingerprint.
- A cancellation request removes a queued job and signals an active worker to terminate its SSH channel/remote process. The final cancellation event is auditable and no later worker event may overwrite a terminal cancellation state.

### Directory to scan safely

The managed-machine scan control exposes the same **directory target** concept as README Option 1’s scanner `--target` argument. It selects the directory scope on the current managed machine for a newly launched scan.

- The UI calls this field **Directory to scan** to avoid confusing it with the managed-machine record.
- The default remains the managed machine’s approved root directory (`.` unless a policy-approved default is later supplied).
- The selected directory is validated as a non-empty, bounded path value before a new job is created and is stored only in that run’s immutable runtime context.
- It never changes the managed target ID, target machine, or directory of an existing run.
- Existing evidence remains permanently attributed to its original machine and directory scope.
- If a scan is active, the directory cannot be changed for that run. The UI explains that the user must stop it, choose a directory, then launch a new scan.
- The application must not use this field to execute shell text; the worker passes it only as the scanner program's structured `--target` argument.

## Authorization and API contracts

- Scan launch remains allowed for `admin`, `operator`, and `analyst`.
- Stop/cancel requires `scan_jobs_live_progress:edit`, which permits `admin` and `operator`; analysts retain launch-only permission.
- API authorization is authoritative. Next.js proxy role checks are defense in depth.
- Add a tenant-scoped cancellation endpoint for an individual scan run, with a matching same-origin proxy allowlist.
- Accept an optional `scan_target` directory only on new managed-machine scan-job creation. Validate it as data, persist it in the run context, and pass it to the scanner as a discrete `--target` argument.
- Cancellation rejects unknown, cross-tenant, and terminal runs without modifying a different run or target.
- Public scan DTOs remain limited to ID, status, timestamps, target ID, user-safe events, the selected scan directory, and browser-safe hardware summary.

## UI behavior

1. A machine opens with its identity, hardware line, state, and scan status visible without expanding settings.
2. Before launching a scan, authorized users can set **Directory to scan**; the field defaults to `.` and is shown in the queued/run summary after launch.
3. An active scan displays progress, current state, **Stop scan**, and the immutable directory scope.
4. Clicking **Stop scan** disables duplicate actions while the request is pending, then refreshes the run history and re-enables directory selection and **Scan machine** after confirmed cancellation.
5. Directory selection remains disabled while a scan is active; its helper text explains that a fresh scan is required for a new directory.
6. Disabled actions explain why they are unavailable; no destructive browser confirmation is used for stopping a scan because cancellation preserves all existing evidence.

## Testing and verification

Use strict TDD in vertical slices:

1. cancellation API and worker terminal-state behavior;
2. same-origin cancellation authorization/proxy behavior;
3. client scan control states and directory-scope launch behavior;
4. scanner `--target` argument propagation and safe path validation;
5. browser-safe hardware data mapping and fallback;
6. copy and typography regression checks.

Release gates:

- focused API, worker, web, and Python regressions;
- full repository test/check/build gates;
- production Docker build;
- authenticated browser QA at desktop, tablet, and mobile with a controlled active-scan fixture;
- independent security/QA review;
- updated changelog/version files and annotated `v1.1.0` tag;
- rollback backup, staged live deployment, mounted-route/API/log verification.

## Out of scope

- Retargeting an existing run.
- Deleting active scans, history, findings, or artifacts.
- Changing the one-time-audit product direction.
- Persisting raw scanner or hardware inventory payloads in browser-visible machine data.
- Broad application redesign outside the managed-machine workflow and shared typography corrections required for consistency.
