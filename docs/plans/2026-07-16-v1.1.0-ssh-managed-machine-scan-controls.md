# SSH Managed-Machine Scan Controls Implementation Plan

> **For Hermes:** Use `subagent-driven-development` to execute this plan task-by-task with spec-compliance review followed by code-quality review.

**Goal:** Release Shore Sentinel `v1.1.0` with authenticated SSH-push scans on managed machines, explicit directory scope, cancellable scan runs, browser-safe hardware summaries, and polished machine-page copy and typography.

**Architecture:** The browser submits a bounded directory path only when creating a new scan. The API validates and persists this immutable scope, then queues a job whose ID is also the BullMQ ID. An internal-token-authenticated worker requests a short-lived execution context, verifies a pinned host key, transfers the packaged scanner bundle over SFTP, and launches the remote scanner with the directory passed as one safely encoded argument. The worker reports events/artifacts through authenticated internal routes and honours Redis cancellation messages.

**Tech Stack:** Next.js, NestJS, PostgreSQL JSONB, BullMQ/Redis, `ssh2`, Python 3 scanner bundle, SFTP, Docker Compose, Playwright.

**Hard security invariants:**

- Browser payloads never contain SSH credentials or host key material.
- A `scan_target` is data, not shell syntax: reject NUL, newline/control characters, excessive length, and relative traversal segments except the default `.`.
- SSH host verification is mandatory; no trust-on-first-use behavior.
- Internal worker routes require `SHORE_SENTINEL_WORKER_TOKEN`; no default development secret is accepted outside test mode.
- Cancellation is terminal: no later lifecycle event can change a cancelled run/job.
- `runtime_context` stores only safe `scan_target` and hardware summary fields; it never stores credentials or raw scanner output.
